# Feature Specification: 課程作業與批改系統

**Feature Branch**: `010-lesson-homework`  
**Created**: 2026-05-09  
**Status**: Draft  
**Input**: 在小節影片/內文下方顯示 Markdown 作業題目，學員可提交作業留言（最多2層巢狀），老師可批改回覆；管理後台設有作業批改專區，管理題目並一次批改同一小節所有學員作業。

---

## Clarifications

### Session 2026-05-09

- Q: 管理員回覆學員作業後，學員如何得知？ → A: 在導覽列新增通知鈴鐺（非完整通知中心），點擊展開最近 5 則站內通知，格式為「講師已回覆您在【課程名稱】的作業/提問！」，點擊後跳轉至對應小節教室並清除未讀。
- Q: 管理員在後台對作業題目執行「移除」操作後，學員提交的資料如何處理？後台是否仍顯示？ → A: 不做永久刪除，改為「下架」操作——題目與所有回覆資料完整保留於資料庫，後台批改專區仍可查看；前台完全不顯示任何痕跡（不顯示「已刪除」標記）。下架的題目可在後台重新上架。
- Q: 學員在教室頁面的提交記錄，排序方向為何？ → A: 舊到新（時間正序）——最早的提交在上方，新的追問與批改自然往下增長，符合對話閱讀習慣。
- **增量更新 2026-05-09**: 新增作業完成標記（US-7）、積分制與勳章（US-8）。管理員可在批改專區對每位學員每道題標記一次「已完成」，觸發 +100 積分、完成通知與教室勳章。積分記錄在 User 帳號，起始為 0，只增不減；兌換功能為未來規劃。
- Q: 管理員是否需要手動調整學員積分的功能？ → A: 本期不提供，積分僅由系統在「標記完成」時自動累計，無手動增減或撤銷機制。
- Q: 學員點擊通知時，若該作業題目已被下架，應如何處理？ → A: 仍跳轉至對應小節的教室頁面，作業區依正常邏輯不顯示（因題目下架），通知標記為已讀，不顯示任何錯誤訊息。
- Q: 帳號設定頁積分顯示詳細度為何？後台是否可查看學員完成記錄？ → A: 顯示總分 + 完成記錄列表（課程名稱、小節名稱、+100、完成日期，由新到舊）；後台會員管理頁面也要能查看每位會員的作業完成紀錄。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 學員提交作業 (Priority: P1)

已購買課程並登入的學員，在上課教室觀看影片或閱讀內文後，可在同一頁面下方看到作業題目（Markdown 渲染），並透過文字輸入框提交自己的作業答案或問題。試閱學員看不到作業區。

**Why this priority**: 作業提交是整個系統的核心入口，沒有提交功能，批改功能就無從施展。

**Independent Test**: 以已購買的學員帳號進入有作業的小節，確認頁面下方顯示題目區，能成功送出文字，頁面即時顯示提交記錄。

**Acceptance Scenarios**:

1. **Given** 學員已登入且已購買課程，進入有作業題目的小節，**When** 頁面載入完成，**Then** 小節內容下方出現作業題目區，以 Markdown 正確渲染題目內容。
2. **Given** 學員在作業區以 Markdown 格式輸入內容並點擊送出，**When** 送出成功，**Then** 新提交的內容以 Markdown 渲染顯示在留言列表中，輸入框清空。
3. **Given** 學員以「試閱」模式（未購買）瀏覽小節，**When** 頁面載入，**Then** 作業題目區不顯示。
4. **Given** 學員未登入，**When** 進入課程，**Then** 無法看到作業區（被登入牆擋住）。

---

### User Story 2 - 學員管理自己的提交記錄 (Priority: P2)

學員可查看自己在某小節提交過的所有作業記錄（包含老師的回覆）。學員可以編輯或刪除自己發出的留言，但看不到其他學員的提交內容。

**Why this priority**: 學員在提交後需要能追蹤批改狀態並修正自己的作業，是完整學習閉環的必要部分。

**Independent Test**: 提交作業後，確認只顯示自己的記錄；嘗試編輯一筆提交，確認內容更新；刪除一筆，確認移除。

**Acceptance Scenarios**:

1. **Given** 學員曾多次提交作業，**When** 重新進入該小節，**Then** 只顯示本人的提交記錄（按提交時間由舊到新排列），並附帶老師的回覆（若有）。
2. **Given** 學員點擊某筆留言的「編輯」，**When** 修改內容並儲存，**Then** 留言內容更新，並顯示「已編輯」標記。
3. **Given** 學員點擊某筆留言的「刪除」，**When** 確認刪除，**Then** 該留言從列表移除；若有老師回覆，連同回覆一併刪除。
4. **Given** 老師已回覆學員的提交，**When** 學員查看該提交，**Then** 老師回覆緊接顯示在提交下方，且學員可在回覆下方追加補充（不超過第2層）。

---

### User Story 3 - 管理員建立與管理作業題目 (Priority: P1)

管理員可在後台「作業批改專區」為任意小節建立、修改或刪除作業題目，題目以 Markdown 格式撰寫，支援即時預覽。

**Why this priority**: 沒有題目就沒有作業，這是整個功能的基礎設置步驟。

**Independent Test**: 在後台選擇一個小節，建立題目，確認前台教室正確顯示；修改題目後確認前台同步更新；刪除題目後確認前台不再顯示作業區。

**Acceptance Scenarios**:

1. **Given** 管理員進入作業批改專區，**When** 選擇一個尚無題目的小節，**Then** 顯示「新增題目」按鈕，點擊後展開 Markdown 編輯器與預覽區。
2. **Given** 管理員在 Markdown 編輯器輸入題目並儲存，**When** 學員進入該小節，**Then** 作業題目正確以 Markdown 渲染顯示。
3. **Given** 管理員對已有題目的小節點擊「編輯」，**When** 修改並儲存，**Then** 前台學員端立即呈現更新後的題目。
4. **Given** 管理員點擊「下架題目」，**When** 確認下架，**Then** 該小節作業區在前台完全消失（無任何殘留提示），題目與所有學員提交資料在後台保留，並可重新上架。

---

### User Story 4 - 管理員批改學員作業 (Priority: P1)

管理員在作業批改專區可一次性瀏覽某小節所有學員的作業提交，並針對每筆提交進行 inline 回覆（批改）。頁面預設顯示所有課程、所有小節的最新提交，可依課程、小節篩選。

**Why this priority**: 批改功能是管理員與學員互動的核心，也是這個功能模組對教學品質的主要貢獻。

**Independent Test**: 以管理員身份進入批改專區，確認看到所有學員的提交；選擇特定課程和小節後，確認只顯示該小節的提交；對一筆提交送出回覆，確認在學員端也能看到該回覆。

**Acceptance Scenarios**:

1. **Given** 管理員進入作業批改專區，**When** 頁面載入，**Then** 顯示所有小節中最新學員提交，按提交時間倒序排列，每筆顯示學員姓名、提交內容與時間。
2. **Given** 管理員從課程下拉選單選擇特定課程，再從小節下拉選擇特定小節，**When** 篩選套用，**Then** 只顯示該小節的題目及所有學員提交，其他小節不顯示。
3. **Given** 管理員在某筆學員提交旁點擊「回覆」，**When** 輸入批改內容並送出，**Then** 回覆立即出現在該提交下方；學員進入教室時可看到該回覆。
4. **Given** 尚無任何學員提交，**When** 管理員查看該小節，**Then** 顯示「尚無學員提交」提示。

---

### User Story 6 - 學員收到作業批改通知 (Priority: P2)

登入後的學員，在頁面頂端導覽列看到通知鈴鐺圖示。當老師回覆了某筆作業提交後，通知鈴鐺出現紅點與未讀數字；點擊展開顯示最近 5 則通知訊息；點擊通知後跳轉至對應小節的教室頁面，並清除該通知的未讀狀態。

**Why this priority**: 學員在提交作業後無從得知批改進度，通知機制讓學習閉環完整，但不影響核心提交/批改流程，故為 P2。

**Independent Test**: 管理員回覆一筆學員作業後，以該學員帳號登入，確認導覽列通知鈴鐺出現紅點且數字為 1；點擊通知鈴鐺，確認展開清單顯示「講師已回覆您在【XXX課程】的作業/提問！」；點擊該通知，確認跳轉至正確的教室小節，且通知標記為已讀（紅點消失）。

**Acceptance Scenarios**:

1. **Given** 老師在批改專區回覆了學員的提交，**When** 該學員下次登入並查看任何頁面，**Then** 導覽列通知鈴鐺顯示紅點及未讀數字（該數字等於未讀通知數）。
2. **Given** 學員點擊通知鈴鐺，**When** 展開通知下拉列表，**Then** 顯示最近最多 5 則通知，每則格式為「講師已回覆您在【課程名稱】的作業/提問！」及時間戳。
3. **Given** 學員點擊某則通知，**When** 導航發生，**Then** 跳轉至該通知對應的課程小節教室頁面，且該通知標記為已讀（不再計入未讀數）。
4. **Given** 學員已讀取所有通知，**When** 查看鈴鐺，**Then** 紅點與數字消失，通知列表仍可展開查看歷史（最多 5 則）。
5. **Given** 學員尚未有任何通知，**When** 點擊鈴鐺，**Then** 展開顯示「目前沒有通知」提示。

---

### User Story 7 - 管理員標記學員作業「已完成」並發放積分 (Priority: P2)

管理員在批改專區瀏覽某位學員的作業提交後，可對該學員在這道題目上的整體作答按下「標記已完成」。系統立即為該學員增加 100 積分，並發送完成通知。每位學員每道題只能被標記一次；已標記者顯示「已完成」狀態，不顯示再次標記的按鈕。

**Why this priority**: 積分獎勵是促進學員積極交作業的核心激勵機制，但需先有提交與批改功能才有意義，故列 P2。

**Independent Test**: 管理員在批改專區找到某學員的某道題提交，點擊「標記已完成」，確認：(1) 該學員積分增加 100；(2) 學員收到完成通知；(3) 批改專區顯示「已完成」狀態；(4) 再次嘗試標記同一學員同一題目時按鈕不可用或不存在。

**Acceptance Scenarios**:

1. **Given** 管理員在批改專區查看某學員對某道題的提交，且該學員尚未被標記為完成，**When** 點擊「標記已完成」並確認，**Then** 該學員的積分立即增加 100，批改專區的該學員區塊顯示「✓ 已完成」標籤，標記按鈕消失。
2. **Given** 管理員完成標記，**When** 系統處理，**Then** 自動為該學員建立一則完成通知（格式：「恭喜！您在【課程名稱】的作業已通過，積分 +100！」），並跳轉至對應小節。
3. **Given** 管理員嘗試對同一學員同一道題目再次標記，**When** 查看批改專區，**Then** 該學員的區塊只顯示「✓ 已完成」，無「標記已完成」按鈕，無法重複標記。
4. **Given** 一道題目下有多位學員提交，**When** 管理員標記其中一位學員為完成，**Then** 只有該學員的積分增加，其他學員不受影響。

---

### User Story 8 - 學員查看積分與作業完成勳章 (Priority: P2)

學員在帳號設定頁可查看目前累積的積分總數。教室頁面中，已被標記「已完成」的小節，其作業題目區塊右上角顯示綠色打勾勳章，提供視覺成就感。

**Why this priority**: 積分與勳章是積分制的視覺呈現，讓學員感受到學習進度與獎勵。

**Independent Test**: 完成一筆作業標記後，學員進入帳號設定查看積分是否為 100（假設從 0 開始）；進入有完成標記的小節，確認作業區塊右上角出現綠色打勾勳章。

**Acceptance Scenarios**:

1. **Given** 學員在帳號設定頁，**When** 查看積分區塊，**Then** 顯示累積總分，以及完成記錄列表（課程名稱、小節名稱、+100、完成日期），由新到舊排列；新帳號顯示 0 分與空列表。
2. **Given** 管理員已將某學員在某小節的作業標記為完成，**When** 該學員進入該小節的教室頁面，**Then** 作業題目區塊右上角顯示綠色打勾勳章。
3. **Given** 學員在尚未完成的小節查看作業，**When** 頁面載入，**Then** 題目區塊不顯示勳章。
4. **Given** 學員累積多筆完成記錄，**When** 查看帳號設定，**Then** 積分總數正確，記錄列表逐筆列出每道完成的作業（課程名稱、小節名稱、+100、日期）。
5. **Given** 管理員在後台會員管理頁查看某位會員的詳情，**When** 查看該會員，**Then** 能看到該會員的作業完成記錄列表（課程名稱、小節名稱、完成日期）及積分總計。

### User Story 5 - 管理員管理所有留言 (Priority: P2)

管理員可編輯或刪除平台上任何人的留言（包含學員提交與老師回覆），以處理不當內容或修正錯誤批改。

**Why this priority**: 內容管理權限是平台健康運作的保障，優先級次於核心提交與批改流程。

**Independent Test**: 以管理員身份找到任意一筆學員留言，進行編輯並儲存；確認內容更新；再刪除該筆，確認移除。

**Acceptance Scenarios**:

1. **Given** 管理員在批改專區看到一筆學員留言，**When** 點擊「編輯」並修改後儲存，**Then** 留言內容更新，顯示「已編輯」標記。
2. **Given** 管理員刪除一筆頂層學員提交，**When** 確認刪除，**Then** 該提交及其下所有回覆一併從系統移除。
3. **Given** 管理員刪除一筆回覆（第2層），**When** 確認刪除，**Then** 只移除該回覆，頂層提交保留。

---

### Edge Cases

- 管理員將題目「下架」後——題目與所有提交資料在後台完整保留，管理員批改專區仍可查看；前台完全不顯示作業區塊，任何學員（包含新進入者）都不會看到任何殘留。下架後可重新上架，前台恢復顯示。
- 學員刪除自己的頂層提交時，其下已有老師回覆——需一併刪除，並提示「刪除後老師批改也將消失，確認刪除？」。
- 學員試圖直接存取另一學員的提交——系統回傳 403，不洩露他人資料。
- 同一小節有多位學員大量提交時，批改專區分頁顯示，避免頁面過重。
- 題目內容含有大量 Markdown（表格、圖片連結等）——前台正確渲染，不破壞版面。
- 非管理員用戶嘗試存取他人留言的編輯/刪除 API——系統回傳 403。
- 管理員嘗試重複標記同一學員同一道題為完成——系統拒絕（後端驗證），積分不重複累計。
- 作業完成標記後，該題目被管理員下架——完成勳章不再顯示（因作業區整體隱藏），但學員積分不扣回，完成記錄在後台保留。
- 積分為整數，不因任何操作減少（本期無扣分機制）。
- 通知的跳轉邏輯不檢查作業題目的上架狀態：點擊通知一律跳轉至對應小節，作業區是否顯示由小節頁面自身的顯示邏輯決定。

## Requirements *(mandatory)*

### Functional Requirements

**學員端**

- **FR-001**: 系統必須在已購買課程的學員上課頁面，對有作業題目的小節，於影片/內文下方顯示作業題目區塊。題目以 Markdown 渲染，區塊採用獨立的基礎 CSS 樣式（不沿用課程內文排版），至少正確呈現以下元素：`###`/`####` 標題、有序與無序清單、超連結、引用區塊（blockquote）、粗體。
- **FR-002**: 系統必須允許已登入且具有課程存取權的學員，以 Markdown 格式撰寫並提交作業回覆；提交後以 Markdown 渲染顯示（與題目渲染方式一致）。
- **FR-003**: 系統必須只向學員顯示其本人的提交記錄，按提交時間由舊到新排列，不顯示其他學員的提交。
- **FR-004**: 系統必須允許老師（管理員）在學員提交下方添加回覆，學員可查看該回覆。
- **FR-005**: 系統必須允許學員編輯自己的留言，編輯後顯示「已編輯」標記。
- **FR-006**: 系統必須允許學員刪除自己的留言；若該留言有子回覆，連同子回覆一起刪除，刪除前顯示確認提示。
- **FR-007**: 系統不得向試閱模式（未購買）的使用者顯示作業區。
- **FR-008**: 留言巢狀結構最多 2 層（頂層提交 + 一層回覆），不允許對回覆再進行回覆。

**管理後台**

- **FR-009**: 系統必須提供獨立的「作業批改專區」後台頁面，與現有課程/章節管理分開，並在側邊導覽列中顯示入口。
- **FR-010**: 管理員必須能為任意小節建立、修改作業題目（Markdown 格式，含即時預覽），以及將題目「下架」（停止前台顯示，資料保留）或「重新上架」（恢復前台顯示）。系統不提供永久刪除題目的功能。
- **FR-011**: 批改專區預設顯示所有小節的學員提交（按時間倒序），支援依課程、依小節篩選。
- **FR-012**: 管理員必須能對任意學員提交進行 inline 回覆（批改）。
- **FR-013**: 管理員必須能編輯或刪除平台上任何人的留言。
- **FR-014**: 刪除有子回覆的頂層提交時，必須連同子回覆一起刪除。
- **FR-015**: 批改專區在提交數量多時以分頁呈現，每頁預設 20 筆。

**作業完成標記與積分**

- **FR-020**: 管理員必須能在批改專區，針對特定學員的特定作業題目，點擊「標記已完成」，完成後該按鈕消失，顯示「✓ 已完成」狀態。
- **FR-021**: 每位學員每道作業題目只能被標記為完成一次，系統必須在後端拒絕重複標記。
- **FR-022**: 標記完成時，系統必須自動為該學員增加 100 積分（寫入使用者記錄，新帳號起始為 0）。
- **FR-023**: 標記完成時，系統必須自動為該學員建立一則完成通知，格式：「恭喜！您在【課程名稱】的作業已通過，積分 +100！」。
- **FR-024**: 作業已被標記完成的小節，教室頁面的作業題目區塊右上角必須顯示綠色打勾勳章。
- **FR-025**: 學員必須能在帳號設定頁查看積分總數，以及完成記錄列表（課程名稱、小節名稱、+100、完成日期），由新到舊排列。
- **FR-026**: 後台會員管理頁面的個別會員詳情中，必須顯示該會員的作業完成記錄列表（課程名稱、小節名稱、完成日期）及積分總計。（跨功能依賴：Feature 003 會員管理）

**通知系統**

- **FR-016**: 管理員回覆學員作業提交後，系統必須自動為該學員建立一則站內通知。
- **FR-017**: 通知鈴鐺圖示必須顯示於已登入學員的導覽列，有未讀通知時顯示紅點及未讀數字。
- **FR-018**: 點擊通知鈴鐺時展開下拉列表，顯示最近最多 5 則通知（含回覆通知與完成通知），加時間戳；點擊通知後跳轉至對應小節教室，並將該通知標記為已讀。回覆通知格式：「講師已回覆您在【課程名稱】的作業/提問！」；完成通知格式：「恭喜！您在【課程名稱】的作業已通過，積分 +100！」。
- **FR-019**: 通知僅發送給被回覆的學員本人，其他學員不可見。管理員不顯示通知鈴鐺。

### Key Entities

- **作業題目（Assignment）**: 隸屬於某個小節，每個小節最多一道題，包含 Markdown 格式的題目內容與上架/下架狀態。下架後前台完全不顯示；系統不提供永久刪除，資料永遠保留在後台供管理員查看與重新上架。
- **提交留言（Comment）**: 由學員或管理員發出，隸屬於某道作業題目。分為頂層提交（`parent` 為空，代表學員作業）與回覆（`parent` 指向頂層提交，代表批改或追問）。最多 2 層。
- **小節（Lesson）**: 現有實體，每個小節可選擇性地關聯一道作業題目。
- **使用者（User）**: 學員只能看到自己的提交；管理員可查看和管理所有提交。
- **通知（Notification）**: 由系統自動建立，觸發時機有二：(1) 管理員回覆學員提交；(2) 管理員將學員作業標記為完成。記錄通知類型、未讀狀態、來源課程名稱、小節 ID（供跳轉），最多保留最近 5 則供展示。
- **完成標記（AssignmentCompletion）**: 記錄某位學員在某道作業題目上已完成的事實，每對 (學員, 作業題目) 唯一，不可重複建立。
- **積分（Points）**: 隸屬於使用者帳號，整數值，起始為 0，每次作業標記完成後加 100，只增不減。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 有作業的小節，學員進入頁面後可在 2 秒內看到題目，無需額外操作。
- **SC-002**: 學員提交作業後，內容在 2 秒內顯示於頁面，無需手動重新整理。
- **SC-003**: 管理員在批改專區的課程/小節篩選操作，結果在 2 秒內更新。
- **SC-004**: 100% 的學員提交通過隱私保護——學員 A 無法透過任何介面看到學員 B 的提交內容。
- **SC-005**: 管理員能在單一頁面完成「選課 → 選小節 → 查看所有提交 → 回覆批改」的完整流程，不需跳頁。
- **SC-006**: 作業題目卡片的 `###`/`####` 標題、有序/無序清單、超連結、引用區塊、粗體等元素均正確渲染且樣式獨立，不與課程內文樣式混用；學員提交的 Markdown 內容同樣正確渲染，不破壞頁面版面。
- **SC-007**: 管理員回覆後，學員下次開啟任意頁面即可在導覽列看到通知紅點，無需重新整理教室頁面。
- **SC-008**: 管理員標記完成後，學員的帳號設定頁積分與完成記錄列表立即反映增加（下次開啟帳號設定時顯示正確數值與新增明細）。
- **SC-009**: 積分計算100% 準確——不重複計算（同一道題只加一次），不遺漏（每次標記必加）。

## Assumptions

- 作業提交支援 Markdown 格式撰寫與渲染；本期不支援附件上傳。
- 管理員回覆等同「老師批改」，不區分不同角色的老師身份，平台目前只有一種管理員角色。
- 刪除包含子回覆的頂層留言時，採用連帶刪除策略。
- 小節若無作業題目、或題目處於「下架」狀態，前台完全不顯示作業區塊（不顯示任何提示或殘留）。
- 批改專區的提交列表在提交數量龐大時需分頁，每頁預設顯示 20 筆提交。
- 學員無法看到老師以外的其他學員是否也提交了作業（完全隔離）。
- 通知採站內輕量通知（導覽列鈴鐺），不發 Email 通知，每位學員最多展示最近 5 則。
- 通知僅在管理員回覆時觸發，學員互相追問不觸發。若有多次回覆，每次回覆各建立一則通知。
- 完成標記功能本期不支援「取消完成」、「撤銷積分」或任何形式的管理員手動調整積分，積分只增不減。
- 積分未來可拿來兌換特定商品，但兌換功能不在本期範圍內。
- 積分以整數儲存在使用者帳號中，全平台共用一組積分（不區分課程）。
