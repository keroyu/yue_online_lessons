## Clarifications

### Session 2026-06-09

- Q: 折扣碼驗證 API 是否需要防暴力猜碼機制？ → A: 同一 IP 連續失敗 N 次後節流（Option A）
- Q: 折後金額過低時（金流最低金額限制）如何處理？ → A: 折後金額下限為 NT$1；低於此值時顯示錯誤，不建立免費結帳流程（Option B）
- Q: 折扣碼刪除語義為何？刪除後相同代碼是否可重建？ → A: 軟刪除；代碼從列表隱藏但永久保留，相同字串不可重建（Option B）
- 規劃階段補充（2 項，使用者需求）：統計時間範圍新增「全部」選項（FR-017、US4）；支援銷售頁網址參數 `?coupon=CODE` 自動帶入折扣碼，沿用既有 traffic_source session 機制，手動輸入優先、無效靜默忽略（FR-020、US5）。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 購物車套用折扣碼 (Priority: P1)

已登入或訪客用戶在購物車頁面輸入折扣碼並套用，系統即時驗證並顯示折扣金額與更新後的訂單合計；用戶可移除折扣碼恢復原價。整個操作在前往結帳前完成，不需要跳轉頁面。

**Why this priority**: 這是用戶感知折扣效果的唯一入口。沒有此功能，折扣碼即便建立了也無法被使用，是整個系統的核心互動點。

**Independent Test**: 在購物車中放入一門課程，輸入有效折扣碼點擊套用，確認折扣行出現、合計正確更新；再點移除，確認恢復原價。

**Acceptance Scenarios**:

1. **Given** 購物車中有課程，**When** 用戶輸入有效且未過期的折扣碼並點擊「套用」，**Then** 頁面顯示折扣標籤（如「六折優惠」或「折抵 NT$200」）、折扣金額行（-NT$ XXX）及更新後合計；輸入區變為「已套用的代碼 + 移除按鈕」。
2. **Given** 已套用折扣碼，**When** 用戶點擊「移除」，**Then** 折扣行消失，合計恢復原價，輸入框可重新輸入新代碼。
3. **Given** 購物車中有課程，**When** 用戶輸入不存在或已停用的折扣碼，**Then** 顯示紅色錯誤訊息「折扣碼無效」，合計不變。
4. **Given** 購物車中有課程，**When** 用戶輸入已過期的折扣碼，**Then** 顯示錯誤訊息「折扣碼已過期」。
5. **Given** 購物車中有課程，**When** 用戶輸入已達使用上限的折扣碼，**Then** 顯示錯誤訊息「折扣碼已達使用上限」。
6. **Given** 折扣碼限定特定課程，且購物車中不含該課程，**When** 用戶套用此折扣碼，**Then** 顯示錯誤訊息「此折扣碼不適用於購物車中的課程」。
7. **Given** 折扣碼限定特定課程，且購物車中包含該課程，**When** 用戶套用，**Then** 折扣正常生效。
8. **Given** fixed 類型折扣碼套用後實付金額低於 NT$1，**When** 套用，**Then** 系統拒絕套用並顯示錯誤「折扣金額超過訂單上限，無法套用」，合計維持原價不變。
9. **Given** 已套用折扣碼，**When** 用戶從購物車移除了折扣碼所限定的課程，**Then** 系統自動清除已套用的折扣碼並顯示提示「折扣碼已移除，因限定課程已從購物車中移除」。
10. **Given** 用戶於課程銷售頁點「直接購買」直接進入結帳頁（略過購物車），**When** 進入結帳頁，**Then** 結帳頁同樣提供折扣碼輸入欄可即時套用/移除，與購物車頁行為一致（輸入欄不限於購物車頁）。
11. **Given** 用戶在結帳頁已套用折扣碼後又按「移除」，**When** 送出付款，**Then** 系統不帶入該折扣碼（僅送出 UI 上實際處於「已套用」狀態的代碼），訂單以原價建立。

### User Story 2 - 後台管理員建立與管理折扣碼 (Priority: P1)

管理員在後台新增折扣碼，設定代碼名稱、折扣類型（固定金額或折數）、適用課程、到期日、使用名額，並可隨時啟用或停用；後台列表可一覽所有折扣碼的狀態與使用情況。

**Why this priority**: 無法建立折扣碼，整個系統就無從運作。管理員需要完整的 CRUD 操作來管理優惠活動。

**Independent Test**: 在後台新增一個 fixed 類型折扣碼 "TEST6"（金額 NT$100，全站通用），儲存後在列表中確認出現，前往購物車套用 "TEST6" 確認可用。

**Acceptance Scenarios**:

1. **Given** 管理員在後台折扣碼頁面，**When** 點擊新增，**Then** 可填寫：代碼（1-6 位英數）、折扣類型（固定金額/折數）、折扣值、適用課程（全站通用或指定課程）、到期日（可留空）、使用名額（可留空）、備註說明。
2. **Given** 管理員選擇「固定金額」類型，**When** 填寫折扣值，**Then** 輸入欄顯示 NT$ 金額格式；填寫低於 NT$10 時顯示錯誤「最低折抵金額為 NT$10」。
3. **Given** 管理員選擇「折數」類型，**When** 填寫折扣值，**Then** 輸入欄接受 0.50–0.95 的小數，並即時顯示說明（例：「0.6 = 六折，用戶實付原價 60%」）；超出範圍時顯示錯誤「折數須介於 0.50 至 0.95 之間」。
4. **Given** 管理員嘗試建立與現有折扣碼相同代碼，**When** 送出，**Then** 顯示錯誤「此代碼已存在」，不建立重複記錄。
5. **Given** 折扣碼列表頁，**When** 管理員查看，**Then** 顯示每筆記錄的：代碼、類型與折扣值、適用範圍、到期日（或「永不過期」）、剩餘名額（或「無限制」）、已使用次數、啟用狀態。
6. **Given** 管理員切換某折扣碼的啟用/停用開關，**When** 確認切換，**Then** 該狀態立即生效；用戶在前台套用時系統即時反映最新狀態。
7. **Given** 管理員編輯現有折扣碼，**When** 修改設定並儲存，**Then** 新設定立即生效，列表顯示最新資訊。
8. **Given** 管理員刪除一個折扣碼，**When** 確認刪除，**Then** 該折扣碼從列表消失（軟刪除）；相同代碼字串不可再建立；歷史訂單記錄與統計數據完整保留不受影響。

### User Story 3 - 結帳時折扣生效並記錄 (Priority: P2)

用戶在購物車套用折扣碼後前往結帳，結帳頁顯示折扣摘要；付款完成後，系統以折後金額建立訂單，並在付款確認後更新折扣碼的使用次數。

**Why this priority**: 依賴 US1（套用折扣碼）和現有結帳流程，且金流整合的確認時機（付款 webhook 後）決定了使用次數更新的準確性。

**Independent Test**: 在購物車套用一個名額限制為 1 的折扣碼，完成結帳付款，確認訂單金額為折後金額、使用次數 +1；再用同一折扣碼嘗試結帳，應顯示「已達使用上限」。

**Acceptance Scenarios**:

1. **Given** 購物車已套用折扣碼，**When** 用戶前往結帳頁，**Then** 結帳頁顯示折扣摘要（折扣碼代碼、折扣金額、實付金額），與購物車顯示一致。
2. **Given** 用戶填寫結帳資料並送出付款，**When** 金流確認付款成功，**Then** 訂單記錄：原始金額、折扣金額、實付金額（= 原始 - 折扣）、使用的折扣碼代碼；折扣碼的已使用次數 +1。
3. **Given** 付款失敗或用戶取消付款，**When** 返回購物車，**Then** 折扣碼的使用次數不增加；用戶可重新套用原代碼（若仍有效）。
4. **Given** 結帳送出前，**When** 系統重新驗證折扣碼，**Then** 若折扣碼在填寫表單期間失效（過期/達上限/被停用），系統阻擋付款並提示用戶，要求返回購物車重新選擇。

### User Story 4 - 後台查看折扣碼統計 (Priority: P2)

管理員在後台查看每個折扣碼在指定時間範圍內的使用統計：完成交易筆數、總營收、總折抵金額，以及每筆交易的明細列表，協助評估優惠活動成效。

**Why this priority**: 統計功能是運營決策的依據，但不影響主要購買流程，屬次要功能。

**Independent Test**: 使用一個折扣碼完成 2 筆測試訂單，進入後台該折扣碼的統計頁，確認交易筆數為 2、營收與折抵金額正確，明細列表顯示兩筆資訊。

**Acceptance Scenarios**:

1. **Given** 管理員在後台折扣碼列表，**When** 進入某折扣碼的統計頁，**Then** 顯示時間範圍切換按鈕（最近 7 天 / 30 天 / 60 天 / 90 天 / 全部），預設顯示 30 天；「全部」涵蓋該折扣碼建立以來的所有已完成交易。
2. **Given** 統計頁面，**When** 切換時間範圍，**Then** 完成交易筆數、總營收（折後實付合計）、總折抵金額即時更新。
3. **Given** 統計頁面，**When** 查看交易明細列表，**Then** 每筆記錄顯示：購買者 Email、付款確認時間、結帳金額（折後實付）、原始訂單金額。
4. **Given** 指定期間內無已完成的交易，**When** 查看統計，**Then** 顯示「此期間無交易記錄」，數字均為 0；明細列表為空。
5. **Given** 統計頁面，**When** 查看交易明細，**Then** 僅顯示付款狀態為「已完成」的訂單，待付款或失敗的訂單不列入。

### User Story 5 - 分享連結自動帶入折扣碼 (Priority: P3)

行銷人員可在課程銷售頁網址附加折扣碼參數（例：`/courses/xxx?coupon=SUMMER`）作為分享連結；用戶點擊後，該折扣碼在結帳流程中自動套用，無需手動輸入，提升優惠活動的轉換率。此功能對**已登入與訪客皆適用**（與 US1 一致）：代碼經 session 跨頁保留，購物車載入時由前端以當前購物車內容（登入取伺服器購物車、訪客取 localStorage）呼叫驗證端點自動套用，驗證失敗則靜默忽略。

**Why this priority**: 屬便利性增強功能，依賴 US1（套用邏輯）與既有結帳流程；沒有它，折扣碼仍可手動輸入，故為次要優先。

**Independent Test**: 以 `/courses/{某課程}?coupon=TEST6` 開啟銷售頁，將該課程加入購物車，進入購物車頁，確認 TEST6 已自動顯示為「已套用」狀態且折扣正確；確認可手動移除。

**Acceptance Scenarios**:

1. **Given** 用戶透過帶 `?coupon=CODE` 參數的銷售頁連結進站，**When** 將課程加入購物車並進入購物車頁，**Then** 該折扣碼自動套用並顯示折扣行，狀態與手動輸入一致（含可移除按鈕）。
2. **Given** 網址帶入的折扣碼無效（不存在/過期/停用/不適用），**When** 用戶進入購物車，**Then** 系統靜默忽略該折扣碼，不顯示錯誤訊息，購物車維持原價（落地體驗不受干擾）。
3. **Given** 網址已帶入折扣碼，**When** 用戶手動輸入另一個折扣碼並套用，**Then** 以手動輸入者為準（手動操作優先於自動帶入）。
4. **Given** 自動帶入的折扣碼已套用，**When** 用戶點擊移除，**Then** 折扣清除且不會於重新整理後再次自動套用。

### Edge Cases

- 折扣碼代碼輸入時大小寫不敏感，系統統一轉大寫後比對（用戶輸入 "test6" 等同於 "TEST6"）。
- fixed 折扣金額導致實付金額低於 NT$1 時，系統拒絕套用並顯示錯誤提示，不允許建立 NT$0 或負數訂單（金流不接受此類交易）。
- ratio 折數計算結果四捨五入至整數元（NT$1 單位），避免金流送出小數金額。
- 折扣碼在用戶結帳表單填寫期間才過期或達到上限：系統在送出付款前重新驗證，阻擋並提示用戶。
- 訪客（guest）結帳套用折扣碼：驗證邏輯與已登入用戶相同，用折後金額建立訂單，used_count 在付款確認時更新。
- 折扣碼刪除為軟刪除：代碼從列表消失但資料永久保留，相同代碼字串不可重建，歷史統計與訂單快照完整無缺。
- 管理員修改折扣碼的折扣值（如從 NT$100 改為 NT$200）：已建立的歷史訂單記錄不回溯修改，僅影響修改後的新訂單。
- 多個用戶同時套用同一個名額為 1 的折扣碼：以付款確認（webhook 到達）的順序為準；後確認的訂單其 used_count 超過上限，但付款已完成，系統仍正常開通課程（折扣碼名額為軟限制，高峰並發情況下不強制退款）。
- 折扣碼驗證端點須防止暴力枚舉：同一 IP 連續驗證失敗達到 5 次後，系統暫時拒絕該 IP 的後續嘗試（等待期 60 秒），防止短代碼被自動猜測套用（與 FR-019 一致）。
- 網址帶入折扣碼（`?coupon=`）以與既有來源歸因（UTM/traffic_source）相同的 session 機制暫存，跨「銷售頁→購物車→結帳」導頁保留；訂單建立完成後清除，避免影響下一筆訂單。
- 網址帶入折扣碼於落地頁不立即驗證/提示，僅於購物車載入時驗證；無效則靜默忽略，不干擾瀏覽體驗。
- 統計「全部」期間：不套用 `webhook_received_at` 時間範圍，統計該折扣碼自建立以來所有付款已完成的訂單。
- 折扣碼輸入欄同時存在於購物車頁與結帳頁；「直接購買」直接跳結帳頁時，仍可在結帳頁套用折扣碼（不會因略過購物車而無法使用優惠）。
- 使用次數計數完整性：`used_count` 唯一寫入點為付款確認後的兌現流程（`fulfillOrder`），且以訂單 `status='paid'` 冪等保護避免重複的金流通知/返回造成重複計數；單純按「套用」（僅驗證）、或建立 `pending` 訂單但未付款，均不計數。
- 移除已套用折扣碼後，結帳不得回退使用網址/session 帶入的原始代碼；避免「使用者已移除卻仍被套用並計入使用次數」的情況。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統 MUST 支援管理員建立折扣碼，代碼格式為 1–6 位英數字（不分大小寫，系統統一儲存為大寫），在全系統內唯一。
- **FR-002**: 系統 MUST 支援兩種折扣類型：**fixed**（固定折抵金額，最低 NT$10）和 **ratio**（折數，範圍 0.50–0.95；例：0.60 代表六折，用戶實付原價 60%）。
- **FR-003**: 折扣碼 MUST 可設定適用範圍：「全站通用」（任何課程均可使用）或「指定單一課程」（僅限購物車中含有該課程時可套用）。
- **FR-004**: 折扣碼 MUST 可選填到期日；到期後系統自動拒絕套用，無需管理員手動操作。
- **FR-005**: 折扣碼 MUST 可選填最大使用次數；達上限後系統自動拒絕套用。使用次數以付款成功確認為計，未完成付款不計入。
- **FR-006**: 管理員 MUST 能透過啟用/停用開關即時控制折扣碼的可用狀態；停用後用戶套用時系統立即拒絕，無需重啟或快取清除。
- **FR-007**: 購物車頁面 MUST 提供折扣碼輸入欄位與「套用」按鈕；每次結帳 MUST 僅允許套用一個折扣碼（不支援疊加）。
- **FR-008**: 套用折扣碼成功後，購物車 MUST 顯示：折扣類型標籤（fixed 顯示「折抵 NT$XXX」、ratio 顯示「X折優惠」）、折扣金額行（-NT$ XXX）、更新後實付合計；並提供移除折扣碼的操作。
- **FR-009**: 系統 MUST 在套用折扣碼時執行 server-side 即時驗證，涵蓋：存在性與啟用狀態、有效期限、使用次數上限、課程適用性；驗證失敗時回傳明確分類錯誤訊息。
- **FR-010**: 折扣 MUST 以整筆訂單總金額為基礎計算一次（不依個別課程分別折抵）；折後實付金額最低為 NT$1（不產生 NT$0 或負數訂單）；若折扣計算後低於 NT$1，系統 MUST 拒絕套用並顯示錯誤「折扣金額超過訂單上限，無法套用」；ratio 計算結果四捨五入至整數元。
- **FR-011**: 結帳頁 MUST 顯示折扣摘要（折扣碼代碼、折扣金額、實付金額），資訊與購物車頁一致。
- **FR-012**: 系統 MUST 在送出付款前重新 server-side 驗證折扣碼有效性；若折扣碼在結帳表單填寫期間失效，MUST 阻擋付款並告知用戶原因。
- **FR-013**: 付款成功確認後，系統 MUST 在訂單記錄中儲存：折扣碼代碼、原始金額、折扣金額、實付金額（原始 - 折扣）。
- **FR-014**: 付款成功確認後，系統 MUST 將該折扣碼的已使用次數 +1；付款失敗或取消不計入。
- **FR-015**: 後台 MUST 提供折扣碼列表頁，顯示每筆記錄的：代碼、類型與折扣值、適用範圍、到期日、名額剩餘、已使用次數、啟用狀態。
- **FR-016**: 後台 MUST 支援折扣碼的新增、編輯、停用/啟用、刪除操作。刪除採**軟刪除**：記錄從列表隱藏但保留於資料庫，代碼字串永久佔用（不可重建相同代碼）；歷史訂單的折扣碼欄位與統計數據不受影響。管理員若需暫停優惠應使用「停用」而非刪除。
- **FR-017**: 後台統計頁 MUST 提供時間範圍篩選（最近 7 / 30 / 60 / 90 天 / 全部），切換後數據即時刷新，顯示：完成交易筆數、總營收（折後實付合計）、總折抵金額。「全部」不套用時間範圍限制，統計該折扣碼所有已完成交易。
- **FR-018**: 後台統計頁 MUST 列出指定期間的交易明細，每筆顯示：購買者 Email、付款確認時間、結帳金額（折後）、原始金額；僅顯示付款狀態為「已完成」的訂單。
- **FR-019**: 折扣碼驗證端點 MUST 對同一 IP 實施失敗次數節流：連續失敗 5 次後，該 IP 的後續驗證請求須等待 60 秒才可重試；套用成功後重置失敗計數。
- **FR-020**: 系統 MUST 支援透過課程銷售頁網址參數（`?coupon=CODE`）攜帶折扣碼；用戶進站後該折扣碼在購物車/結帳流程中自動套用並通過 server-side 驗證。手動輸入的折扣碼優先於網址帶入者；網址折扣碼無效時靜默忽略；用戶移除後不再於同一流程自動重套。
- **FR-021**: 折扣碼輸入欄 MUST 同時出現在購物車頁與結帳頁，兩處共用同一驗證與套用邏輯；以涵蓋「直接購買」直接進入結帳、略過購物車的流程，確保任一進入路徑都能套用折扣碼。
- **FR-022**: 結帳送出時 MUST 僅帶入結帳頁 UI 上「實際處於已套用狀態」的折扣碼；若用戶未套用或已移除，MUST NOT 回退使用網址/session 帶入的原始代碼。折扣碼的使用次數（used_count）MUST 僅在「帶折扣碼的訂單付款成功確認」時 +1，按「套用」按鈕或建立未付款訂單皆不計入。

### Key Entities

- **折扣碼 (Coupon Code)**: 唯一代碼（1–6 位大寫英數）、折扣類型（固定金額/折數）與對應值、適用課程（可選，null = 全站通用）、到期日（可選）、使用名額上限（可選）、已使用次數、啟用狀態、備註說明。
- **訂單 (Order)**: 既有實體，擴充記錄：使用的折扣碼代碼、原始訂單金額、折扣金額；實付金額（total_amount）= 原始 − 折扣。
- **購買記錄 (Purchase)**: 既有實體，記錄折扣碼代碼以保留稽核軌跡。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用戶在購物車輸入折扣碼點擊套用後，3 秒內看到結果（成功顯示折扣或明確錯誤提示）。
- **SC-002**: 折扣金額計算與訂單記錄 100% 準確；折後實付金額不出現誤差，ratio 折數四捨五入規則一致。
- **SC-003**: 折扣碼使用次數僅在付款成功確認後計入，付款取消或失敗不影響計數；準確率 100%。
- **SC-004**: 管理員可在 3 分鐘內完成新增一組含完整設定的折扣碼（含代碼、類型、適用範圍、限制條件）。
- **SC-005**: 後台統計頁時間範圍切換後，新數據在 3 秒內完整呈現。
- **SC-006**: 折扣碼停用後，前台套用時立即收到拒絕回應，無延遲或快取問題。
- **SC-007**: 折扣碼被刪除後，歷史訂單的折扣碼代碼欄位與金額資訊保持完整，統計數據不因刪除而遺失。

## Assumptions

- 折扣碼不支援「百分比折扣（off%）」概念，ratio 類型以「折數」（0.50–0.95）呈現，即用戶實付的比例；UI 顯示為「X折」，例：0.6 顯示為「六折優惠」。
- 一筆訂單只能套用一個折扣碼，不支援任何形式的疊加（fixed + ratio、同類型兩個均不行）。
- 折扣碼適用範圍限「全站通用」或「指定單一課程」，不支援「課程分類群組」或「多課程清單」。
- 使用次數統計以「付款確認時間」（金流 webhook 觸發時）為基準，非訂單建立時間。
- 後台統計頁的「總營收」為折後實付金額（而非原始金額），反映實際入帳。
- 管理員後台統計不提供 CSV 匯出（此功能由 006 交易管理模組統一處理）。
- 折扣碼不支援「同一用戶只能使用一次」的個人使用限制；版本一以全局使用次數為唯一限制依據。
- 訪客（guest）結帳可套用折扣碼，驗證規則與已登入用戶完全相同。
- 高峰並發時名額超發屬可接受的邊界情況（軟限制），系統不因此退款或回滾已完成的訂單。
- 網址折扣碼參數固定為 `coupon`（`?coupon=CODE`），僅於課程銷售頁攜帶捕捉；其他頁面不另行捕捉。
- 自動帶入與手動輸入共用同一「已套用折扣碼」狀態與 server-side 驗證，僅來源不同；不支援兩者疊加。
