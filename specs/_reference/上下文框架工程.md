


## 把索引變成 agent 的固定工作流程（navigation protocol）

讓 agent 真正「學會」使用這些較進階的索引方法，關鍵不是只建立 index，而是**把索引變成 agent 的固定工作流程（navigation protocol）**。也就是讓 agent 每次任務都遵循同一套查找順序。實務上通常透過三個元素：repository metadata、檢索工具、以及 agent system prompt。

第一部分是建立 repository metadata。不要只做一個 index.md，而是準備三種結構化資料。
第一個是 repo_map.md。它是整個 repository 的概覽，用非常短的摘要描述每個模組與職責。例如：

```
# Repository Map

Auth System
- login flow
- session management
- oauth integration
files: src/auth/

Billing
- invoice generation
- payment gateway
files: src/billing/
```

LLM 非常擅長先讀 summary 再決定去哪個檔案。repo_map 的作用就是讓 agent 在幾十行文字內理解整個專案。

第二個是 spec_index.json。這個檔案把所有 user story 與 spec 模組對應起來。避免用行號，而是用 anchor 或 heading。

```
{
  "features": [
    {
      "name": "User Login",
      "user_story": "user can login with email/password",
      "file": "spec/auth.md",
      "anchor": "login-flow"
    },
    {
      "name": "Password Reset",
      "user_story": "user can reset password via email",
      "file": "spec/auth.md",
      "anchor": "password-reset"
    }
  ]
}
```

這樣 agent 可以先找 feature，再跳到 spec。

第三個是 vector index（可選）。如果 repo 很大，可以用 embedding 建立 chunk index。簡單做法是把 spec 與 code 切成 500–1000 tokens 的片段，用 embedding model 建立向量搜尋。這層通常只在大型 repo 才需要。

第二部分是提供 agent 的檢索工具。Agent 需要可以呼叫的工具，而不是直接讀整個 repo。常見三個工具：

search_spec
輸入關鍵字，回傳 spec_index.json 裡匹配的 feature。

read_file_section
輸入 file path + anchor，回傳 해당 section。

repo_overview
回傳 repo_map.md。

這三個工具就足以讓 agent 做 navigation。

第三部分是設定 agent 的 navigation protocol。也就是在 system prompt 中規定固定流程。例如：

Agent navigation protocol

1. 先讀 repo_map.md 理解專案結構
2. 使用 spec_index.json 找到對應 user story
3. 只讀相關 spec section
4. 再讀對應 code file
5. 不要掃描整個 repository

這一步其實是整個系統最重要的地方。沒有 protocol，agent 會亂讀檔案，token 很快爆掉。

第四部分是建立 repo map 的自動生成。手動維護會很痛苦，所以通常寫一個 script 自動生成。例如 Python 腳本掃描資料夾與 README，生成 repo_map.md。Spec index 也可以從 markdown heading 解析生成。

一個簡化的流程是：

掃描 spec 目錄
解析所有 ## heading
生成 spec_index.json

這樣 spec 更新時 index 會自動更新。

第五部分是 agent workflow。完整流程通常是：

user query
→ agent call repo_overview
→ agent search spec_index
→ agent read specific spec section
→ agent read code file
→ agent 回答或修改 code

這種 workflow 的好處是 context 幾乎不會浪費在無關檔案。

如果把這套系統放到像 OpenClaw、Claude Code、Cursor 這類 agent 裡，實際效果通常是：
agent 不需要掃整個 repo，就能定位功能。大型專案的效率會提升非常多。

還有一個進階技巧，很多 AI engineer 在用但很少公開。就是在 repo_map.md 裡加「intent summary」。例如：

```
Billing module
purpose: handle all payment flows
main entry: billing_service.py
key objects: Invoice, PaymentSession
```

這會讓 LLM 更容易理解模組語義。


## Spec-Kit 專用的 agent navigation 架構


真正有效的做法不是「再寫一份更長的 index 給 agent 看」，而是把 agent 的工作流程改成：

先看全局地圖，再查 spec 索引，再只讀必要片段，最後才進 code。

這跟現在幾個成熟系統的方向一致。Cursor 會先做 codebase indexing，支援 semantic search；Sourcegraph 除了文字搜尋，還做符號／跨 repo 導航；Augment 則明確把它包成 context engine。重點都不是「把更多文件塞進 context」，而是「先索引、再檢索」。([Cursor](https://cursor.com/help/customization/indexing?utm_source=chatgpt.com "Codebase indexing | Cursor Docs"))

你可以這樣教會 agent。

第一步，不要再把 index.md 當成主體，而是拆成三層。

第一層叫 `repo_map.md`，用途是讓 agent 在 30 秒內理解整個 repo。它只寫模組、責任、入口檔、關聯，不寫長篇敘述。

```md
# Repo Map

## Auth
purpose: login, session, oauth
main_files:
- src/auth/service.ts
- src/auth/controller.ts
related_specs:
- specs/auth.md#login-flow
- specs/auth.md#password-reset

## Billing
purpose: checkout, invoice, webhook
main_files:
- src/billing/service.ts
- src/billing/webhook.ts
related_specs:
- specs/billing.md#checkout
- specs/billing.md#refund
```

第二層叫 `spec_index.json`，用途是讓 agent 用 feature / user story 去找 spec，而不是靠人工猜檔名。

```json
{
  "features": [
    {
      "id": "auth.login",
      "title": "User Login",
      "user_story": "As a user, I can log in with email and password",
      "file": "specs/auth.md",
      "anchor": "login-flow",
      "keywords": ["login", "signin", "auth", "session"],
      "code_files": [
        "src/auth/service.ts",
        "src/auth/controller.ts"
      ]
    }
  ]
}
```

第三層才是可選的 `vector index`。如果 repo 不大，可以先不做。Cursor 自己就會自動索引 codebase；Augment 也支援 workspace / remote indexing。也就是說，如果你已經在用這類工具，先把前兩層做好，常常就已經有感。([Cursor](https://cursor.com/help/customization/indexing?utm_source=chatgpt.com "Codebase indexing | Cursor Docs"))

第二步，要把這三層變成 agent 的硬規則，不是建議。

你需要一份 `AGENT_PROTOCOL.md` 或 system prompt，明文規定 agent 的搜尋順序。像這樣：

```md
# Agent Navigation Protocol

When working in this repository, follow this order strictly:

1. Read repo_map.md first to identify the relevant module.
2. Search spec_index.json by feature, user story, and keywords.
3. Read only the matched spec section via file + anchor.
4. Read only the code files listed in code_files.
5. Expand outward only if implementation evidence shows the first files are insufficient.
6. Do not scan the whole repository unless explicitly requested.
7. Before editing, restate:
   - target feature
   - target spec section
   - target files
```

這一段很重要。因為 agent 不是「不知道 index 存在」，而是「即使知道，也常常不照順序做」。你要把它寫成 protocol，而不是參考說明。

第三步，要讓 agent 每次先輸出一個極短的「定位摘要」，確認它真的有照流程走。格式固定，越短越好：

```md
Target feature: auth.login
Spec section: specs/auth.md#login-flow
Code files:
- src/auth/service.ts
- src/auth/controller.ts
Need more search: no
```

它能強迫 agent 在真正開始讀檔前，先完成一次「定位」。對節省 context 很有效。

第四步，不要用行號，改用 anchor、heading id、或穩定 ID。

你前面和 Agent 討論到行號，判斷是對的。行號會過時。現在比較好的做法是：

在 spec 標題上加穩定 ID，例如：

```md
## Login Flow {#auth-login-flow}
```

或在 JSON 裡直接維護：

```json
"id": "auth.login.flow"
```

然後 agent 永遠靠 `file + anchor` 或 `feature id` 定位，不碰行號。

第五步，把 index 自動生成，不要手維護。

真正能長期用的關鍵，不是格式有多漂亮，而是每次更新 spec 後它會不會一起更新。你應該有一支 script：

`build_repo_index.py`

它做三件事：

1. 掃描 `specs/**/*.md`
    
2. 抽出 heading、user story、feature id、keywords
    
3. 產出 `spec_index.json` 和簡版 `repo_map.md`
    

最小可行規則可以很簡單：

如果看到標題 `##` 就當 section。  
如果看到 `User Story:` 或 `As a ...` 就抓進摘要。  
如果 frontmatter 有 `feature_id` 就直接用。

例如 spec 寫成：

```md
---
feature_id: auth.login
module: auth
code_files:
  - src/auth/service.ts
  - src/auth/controller.ts
keywords:
  - login
  - auth
  - session
---

## Login Flow {#login-flow}

User Story:
As a user, I can log in with email and password.
```

這樣 script 很容易解析。

第六步，對不同 agent 採不同接法。

如果你用的是 Cursor，重點不是自己再造一個 embedding 系統，而是讓 Cursor 的索引有更好的「可檢索目標」。Cursor 本來就會做 codebase indexing 和 semantic search，所以你的 `repo_map.md`、`spec_index.json`、結構良好的 spec，會變成它更容易抓到的高密度導航資料。([Cursor](https://cursor.com/help/customization/indexing?utm_source=chatgpt.com "Codebase indexing | Cursor Docs"))

如果你用的是 Claude Code / OpenClaw 這種比較靠 prompt orchestration 的方式，那你就更需要顯式 protocol，甚至可以在每次任務開頭先執行：

1. 讀 `repo_map.md`
    
2. 搜 `spec_index.json`
    
3. 根據 `code_files` 讀指定檔案
    

也就是把它變成固定 preflight。

第七步，別把 AGENTS.md 寫得太胖。

這點反而是近來很多團隊踩雷的地方。至少有業界近期文章引用研究指出，單純堆很大的 context files，可能不但增加推理成本，還會讓任務成功率下降。這表示你的方向應該是「更精準的導航檔」，不是「更肥的總說明」。([Augment Code](https://www.augmentcode.com/blog/your-agents-context-is-a-junk-drawer?utm_source=chatgpt.com "Your agent's context is a junk drawer"))

所以你應該控制成這樣：

`AGENT_PROTOCOL.md`：只寫規則，不寫大段專案背景。  
`repo_map.md`：只寫模組摘要。  
`spec_index.json`：只寫可檢索結構。  
詳細內容回到 spec 本身。

第八步，給 agent 一套你可以直接複製使用的任務 prompt。

你可以把下面這段當成標準開場：

```md
You are working in a spec-driven repository.

Follow this navigation protocol strictly:
1. Read repo_map.md first.
2. Search spec_index.json for the feature or user story most relevant to the task.
3. Read only the matched spec section.
4. Read only the code files explicitly linked from that spec entry.
5. Do not scan unrelated directories unless needed.

Before making any change, output:
- Target feature
- Spec section
- Target code files
- Whether further search is needed

When uncertain, prefer narrow retrieval over broad reading.
```

如果你想再加強一點，可以再補一句：

```md
If multiple candidate features are found, rank top 3 by relevance and choose one before reading code.
```

這能避免 agent 一口氣把整個 spec 目錄讀掉。

最後，給你一個最務實的落地版本。你現在就先做這四件事，不要一次做太大：

1. 統一 spec 的 heading 與 anchor 命名。
    
2. 補一個 `spec_index.json`，先只收錄 feature、user story、file、anchor、code_files。
    
3. 寫一個很短的 `repo_map.md`。
    
4. 寫 `AGENT_PROTOCOL.md`，要求 agent 每次先輸出定位摘要。
    

這樣就已經能明顯提升效率。因為現在主流工具的強項本來就是索引與檢索，不是死背一大坨背景文件。Cursor 會自動索引 codebase；Sourcegraph 強在符號與導覽；Augment 強調能語意索引並映射大型 codebase。你要做的是讓你的 spec 也變成「可索引的結構化上下文」。([Cursor](https://cursor.com/help/customization/indexing?utm_source=chatgpt.com "Codebase indexing | Cursor Docs"))

如果你要，我下一步可以直接幫你產出一套可用的檔案骨架：`repo_map.md`、`spec_index.json` schema、`AGENT_PROTOCOL.md`，以及自動生成它們的 Python script。


## 產出一套可用的檔案骨架

下面是一套可以直接放進 repository 的最小可行結構。它的目標不是增加文件，而是讓 agent **有一條固定的導航路徑**。整個設計只有四個核心檔案與一個生成腳本。

首先是 repository 的目錄結構。保持簡單。

```
repo/
 ├ specs/
 │   ├ auth.md
 │   ├ billing.md
 │
 ├ src/
 │   ├ auth/
 │   ├ billing/
 │
 ├ repo_map.md
 ├ spec_index.json
 ├ AGENT_PROTOCOL.md
 │
 └ tools/
     └ build_spec_index.py
```

repo_map.md 的作用是讓 agent 在幾十行內理解整個 repo。只寫模組、目的、主要檔案與對應 spec，不寫長段說明。

repo_map.md 範例：

```
# Repository Map

## Auth Module
purpose: user authentication and session management

main_files
- src/auth/service.ts
- src/auth/controller.ts

related_specs
- specs/auth.md#login-flow
- specs/auth.md#password-reset


## Billing Module
purpose: checkout, payment processing, invoice generation

main_files
- src/billing/service.ts
- src/billing/webhook.ts

related_specs
- specs/billing.md#checkout
- specs/billing.md#refund
```

第二個檔案是 spec_index.json。這是 agent 的主要導航表，讓它用 feature 或 user story 找 spec，而不是掃整個資料夾。

spec_index.json 範例：

```
{
  "features": [
    {
      "id": "auth.login",
      "title": "User Login",
      "user_story": "As a user I can login using email and password",
      "file": "specs/auth.md",
      "anchor": "login-flow",
      "keywords": ["login", "auth", "signin"],
      "code_files": [
        "src/auth/service.ts",
        "src/auth/controller.ts"
      ]
    },
    {
      "id": "billing.checkout",
      "title": "Checkout",
      "user_story": "User can pay for a purchase",
      "file": "specs/billing.md",
      "anchor": "checkout",
      "keywords": ["checkout", "payment", "invoice"],
      "code_files": [
        "src/billing/service.ts"
      ]
    }
  ]
}
```

第三個檔案是 AGENT_PROTOCOL.md。這個檔案不是說明文件，而是 **agent 的行為規則**。

AGENT_PROTOCOL.md 範例：

```
Agent Navigation Protocol

When working in this repository follow this order.

1. Read repo_map.md to understand module structure.
2. Search spec_index.json to locate the relevant feature.
3. Read the specific spec section using file + anchor.
4. Read only the code files listed in the spec entry.
5. Expand search only if implementation requires it.

Before editing any code output:

Target feature
Spec section
Target files

Do not scan the entire repository unless explicitly required.
Prefer narrow retrieval over broad reading.
```

第四個部分是 spec 文件本身。為了讓 index 穩定，需要使用 anchor 或 feature id。不要使用行號。

specs/auth.md 範例：

```
---
feature_id: auth.login
module: auth
code_files:
  - src/auth/service.ts
  - src/auth/controller.ts
keywords:
  - login
  - auth
---

## Login Flow {#login-flow}

User Story
As a user I can login with email and password.

Steps
1. User submits credentials
2. Server validates credentials
3. Session token is created
```

接下來是自動生成 index 的腳本。這個腳本會掃描 specs 資料夾並生成 spec_index.json。

tools/build_spec_index.py：

```python
import os
import re
import json
import yaml

SPEC_DIR = "specs"

features = []

for root, dirs, files in os.walk(SPEC_DIR):
    for file in files:
        if not file.endswith(".md"):
            continue

        path = os.path.join(root, file)

        with open(path) as f:
            text = f.read()

        frontmatter = {}
        if text.startswith("---"):
            parts = text.split("---", 2)
            frontmatter = yaml.safe_load(parts[1])

        anchors = re.findall(r"## .*?\{#(.*?)\}", text)

        if "feature_id" in frontmatter:
            features.append({
                "id": frontmatter.get("feature_id"),
                "title": frontmatter.get("feature_id"),
                "user_story": "",
                "file": path,
                "anchor": anchors[0] if anchors else "",
                "keywords": frontmatter.get("keywords", []),
                "code_files": frontmatter.get("code_files", [])
            })

with open("spec_index.json", "w") as f:
    json.dump({"features": features}, f, indent=2)

print("spec_index.json generated")
```

執行：

```
python tools/build_spec_index.py
```

每次新增 spec 或 user story 時，只需要重新生成 index。

最後是 agent 的實際工作流程。理想情況下 agent 的推理順序會是：

1. 讀 repo_map.md
    
2. 搜 spec_index.json
    
3. 找到 feature
    
4. 讀 spec section
    
5. 讀 code_files
    
6. 再決定是否擴展搜尋
    

這種流程有三個效果：

第一，agent 不需要掃描整個 repo。  
第二，context token 會大幅下降。  
第三，spec 與 code 的關係會變得明確。

如果你在做 spec-driven development（Spec-Kit 那種流程），這套結構通常會讓 agent 在大型 repo 的定位速度提高很多。


### 更進階版本： 

自動生成 repo_map.md、加入 dependency graph，以及讓 agent 能用自然語言直接搜尋 feature。這會接近 Cursor / Sourcegraph 那類工具背後的 indexing 架構。

下面是一個更進階但仍然可落地的版本。目標是讓 agent 不只是查 spec，而是能「導航整個 repo」。核心概念是：自動生成三種索引——模組地圖、spec 索引、依賴關係圖。這三者合在一起就形成一個簡化的 codebase knowledge graph。

先看最終的資料結構。建議在 repo 裡新增一個 `.agent_index/` 目錄。

```
repo/
 ├ specs/
 ├ src/
 ├ .agent_index/
 │   ├ repo_map.json
 │   ├ spec_index.json
 │   ├ dependency_graph.json
 │
 ├ AGENT_PROTOCOL.md
 └ tools/
     └ build_repo_index.py
```

repo_map.json 的用途是模組導航。與之前的 markdown 不同，這裡改用 JSON，因為 agent 與工具都更容易解析。

範例：

```
{
  "modules": [
    {
      "name": "auth",
      "purpose": "authentication and session management",
      "paths": ["src/auth"],
      "entry_files": [
        "src/auth/service.ts",
        "src/auth/controller.ts"
      ],
      "related_specs": [
        "specs/auth.md#login-flow"
      ]
    }
  ]
}
```

spec_index.json 與前面版本相似，但增加了 module 與 dependency hints。

```
{
  "features": [
    {
      "id": "auth.login",
      "module": "auth",
      "title": "User Login",
      "user_story": "user can login using email and password",
      "file": "specs/auth.md",
      "anchor": "login-flow",
      "keywords": ["login","auth","signin"],
      "code_files": [
        "src/auth/service.ts",
        "src/auth/controller.ts"
      ]
    }
  ]
}
```

第三個索引是 dependency_graph.json。這個檔案是 repo 的簡化呼叫關係圖。Agent 可以用它理解哪些檔案可能相關，而不必掃整個專案。

範例：

```
{
  "nodes": [
    "src/auth/service.ts",
    "src/auth/controller.ts",
    "src/db/client.ts"
  ],
  "edges": [
    {
      "from": "src/auth/controller.ts",
      "to": "src/auth/service.ts",
      "type": "import"
    },
    {
      "from": "src/auth/service.ts",
      "to": "src/db/client.ts",
      "type": "import"
    }
  ]
}
```

接著是自動生成所有索引的腳本。這個腳本會：

1 掃描 spec  
2 解析 frontmatter  
3 掃描 src 目錄  
4 建立簡單 dependency graph

tools/build_repo_index.py：

```
import os
import re
import json
import yaml

SPEC_DIR = "specs"
SRC_DIR = "src"
OUT_DIR = ".agent_index"

os.makedirs(OUT_DIR, exist_ok=True)

features = []
modules = {}
nodes = []
edges = []

def parse_imports(filepath):
    imports = []
    with open(filepath, "r", encoding="utf8") as f:
        text = f.read()

    matches = re.findall(r'import .* from [\'"](.*?)[\'"]', text)
    for m in matches:
        imports.append(m)

    return imports

for root, dirs, files in os.walk(SPEC_DIR):
    for file in files:
        if not file.endswith(".md"):
            continue

        path = os.path.join(root, file)

        with open(path) as f:
            text = f.read()

        frontmatter = {}
        if text.startswith("---"):
            parts = text.split("---", 2)
            frontmatter = yaml.safe_load(parts[1])

        anchors = re.findall(r"## .*?\{#(.*?)\}", text)

        feature_id = frontmatter.get("feature_id")

        if feature_id:
            features.append({
                "id": feature_id,
                "module": frontmatter.get("module"),
                "file": path,
                "anchor": anchors[0] if anchors else "",
                "keywords": frontmatter.get("keywords", []),
                "code_files": frontmatter.get("code_files", [])
            })

            module = frontmatter.get("module")
            if module:
                modules.setdefault(module, {"paths": [], "entry_files": []})

for root, dirs, files in os.walk(SRC_DIR):
    for file in files:
        if not file.endswith((".ts",".js",".py",".go",".rs")):
            continue

        path = os.path.join(root, file)
        nodes.append(path)

        imports = parse_imports(path)

        for imp in imports:
            edges.append({
                "from": path,
                "to": imp,
                "type": "import"
            })

repo_map = {
    "modules": [
        {
            "name": k,
            "paths": v["paths"],
            "entry_files": v["entry_files"]
        } for k,v in modules.items()
    ]
}

with open(f"{OUT_DIR}/repo_map.json","w") as f:
    json.dump(repo_map,f,indent=2)

with open(f"{OUT_DIR}/spec_index.json","w") as f:
    json.dump({"features":features},f,indent=2)

with open(f"{OUT_DIR}/dependency_graph.json","w") as f:
    json.dump({"nodes":nodes,"edges":edges},f,indent=2)

print("Agent index generated.")
```

執行一次：

```
python tools/build_repo_index.py
```

之後 agent 的導航流程會變成：

1 讀 `.agent_index/repo_map.json`  
2 搜 `.agent_index/spec_index.json`  
3 定位 spec section  
4 讀 code_files  
5 如果需要再查 dependency_graph.json

這其實就是很多 AI coding 系統背後在做的事情：  
先建 repo graph，再做精準 retrieval，而不是把整個 repo 丟進 context。

最後補一個很實用的小技巧。很多團隊會在 repo 根目錄再加一個 **AGENT_ENTRYPOINT.md**。內容只有幾行：

```
Start here.

1 Read .agent_index/repo_map.json
2 Search .agent_index/spec_index.json
3 Locate spec section
4 Read related code files
```

這樣無論是 Claude Code、Cursor 或其他 agent，打開 repo 時都很容易走到正確流程。
